# 📘 Study Notes: How to Gather Project Data

## 🔹 1. Purpose of Data Gathering
- While executing a project, you must track:
  - ✅ Progress
  - ✅ Time
  - ✅ Cost

✅ Goal:
- Understand:
  - What has been completed
  - What remains
  - Whether the project is on track

---

## 🔹 2. Key Data to Collect

### ✅ 2.1 Actual Start Dates
- Record when tasks **actually begin**

✅ Why it matters:
- Delayed starts may indicate:
  - Potential project delay

---

### ✅ 2.2 Actual Work or Duration

- Track:
  - Hours worked OR
  - Duration of tasks

✅ Insight:
- Helps measure:
  - Productivity
  - Resource utilization

⚠️ Note:
- Tracking hours requires effort but improves **accuracy**

---

### ✅ 2.3 Remaining Work

- Estimate:
  - Work still required to complete tasks

✅ Helps determine:
- Remaining duration
- Accuracy of initial estimates

---

### ✅ 2.4 Cost Data

#### Track:
- Labor costs (based on hours worked)
- Additional costs:
  - Travel
  - Training
  - Fees
  - Materials

✅ Ensures:
- Accurate **budget tracking**

---

## 🔹 3. Why Data Collection Is Important

- Provides visibility into:
  - Current project status
- Helps detect:
  - Delays
  - Cost overruns
- Enables:
  - Better decision-making

---

## 🔹 4. Methods for Collecting Data

### ✅ Manual Methods:
- Status updates from team
- Timesheets
- Reports

---

### ✅ Automated Methods:
- Email reporting
- Online forms
- Project management tools (e.g., dashboards)

✅ Benefits:
- Saves time
- Reduces errors
- Improves consistency

---

## 🔹 5. Sample Data Collection Form

### 📋 Project Status Update Form

#### Task Information:
- Task Name:
- Assigned Resource:
- Reporting Period:

---

#### Progress Tracking:
- Actual Start Date:
- Actual Finish Date (if complete):
- % Complete:

---

#### Work Tracking:
- Actual Hours Worked:
- Remaining Hours:

---

#### Cost Tracking:
- Labor Cost:
- Other Costs (travel, materials, etc.):

---

#### Issues & Risks:
- Current Issues:
- Risks identified:

---

#### Comments:
- Notes / Updates:

---

## 🔹 6. Best Practices

- Collect data **regularly** (daily/weekly)
- Keep data:
  - Accurate
  - Up-to-date
- Automate where possible
- Ensure team understands:
  - What data to report
  - How to report it

---

## 🔹 ✅ Key Takeaways
- Data collection is essential for:
  - Monitoring and controlling the project
- Track:
  - Start dates
  - Work performed
  - Remaining effort
  - Costs
- Use automation to improve efficiency
- Accurate data enables:
  - Better decisions
  - Successful project delivery

# 📘 Study Notes: How to Manage Project Change

## 🔹 1. Purpose of Change Management
- Changes are **inevitable** in any project
- A **Change Management Plan** ensures:
  - Changes are handled **systematically**
  - Impact on project is **controlled**

✅ Goal:
- Approve useful changes while keeping project on track

---

## 🔹 2. Standard Change Management Process

## ✅ Step 1: Receive Change Request
- A request for change is submitted

### Actions:
- Assign someone to **evaluate the request**
- Record it in the **Change Request Log**

---

## ✅ Step 2: Evaluate the Change
- Assess impact on:
  - Scope
  - Schedule
  - Cost
  - Resources

✅ Include:
- Recommendation:
  - Approve or Reject

---

## ✅ Step 3: Submit to Change Review Board (CRB)
- Present:
  - Change request
  - Impact analysis
  - Recommendation

---

## ✅ Step 4: Decision

### ❌ If Rejected:
- Inform requester
- Ensure:
  - No work begins on the change

---

### ✅ If Approved:
- Assign an **owner** to manage the change

---

## ✅ Step 5: Implement the Change

### Actions:
- Update **baseline documents**:
  - Schedule
  - Budget
  - Scope
- Add:
  - New tasks (if needed)
  - Adjust timelines and costs

### ✅ Important:
- Tag all updates with:
  - **Change Request Number**

👉 Ensures traceability

---

## ✅ Step 6: Track and Report Progress

- Change owner must:
  - Update **Change Request Log**
  - Monitor progress
  - Report status regularly

---

## ✅ Step 7: Close the Change Request

- Once complete:
  - Update final details in log:
    - Actual hours
    - Actual cost
    - Updated finish date (if applicable)

---

## 🔹 3. Special Cases in Change Management

### ✅ 3.1 Impact on Business Case
- Requires approval from:
  - **Customer**

---

### ✅ 3.2 Impact on Other Projects
- Requires approval from:
  - **Executive Board**

---

### ✅ 3.3 Emergency Changes
- When immediate action is required:

### Options:
- Call an **emergency review meeting**
- Seek approval from:
  - Customer or sponsor

---

## 🔹 4. Key Roles in Change Management

- **Evaluator** → Assesses impact  
- **Change Review Board (CRB)** → Approves/rejects changes  
- **Change Owner** → Implements and tracks change  

---

## 🔹 5. Tools and Artifacts

### 📊 Change Request Log:
Tracks:
- Request details
- Status
- Owner
- Progress
- Final results

---

### 📄 Baseline Documents:
- Updated to reflect approved changes
- Maintain:
  - Version control
  - Traceability

---

## 🔹 6. Best Practices

- Never implement changes without approval  
- Communicate decisions clearly  
- Track all changes formally  
- Maintain audit trail using change IDs  
- Act quickly for urgent changes  

---

## 🔹 ✅ Key Takeaways
- Change management keeps projects **controlled and aligned**
- Every change must:
  - Be evaluated
  - Be approved before execution
- Approved changes must update:
  - Baseline documents
- Special approvals are needed for:
  - Major impacts
  - Emergencies
- Proper change tracking ensures:
  - Transparency
  - Accountability
  - Project success

# 📘 Study Notes: Managing Project Scope

## 🔹 1. What Is Project Scope?
- **Scope** defines:
  - What is included in the project
  - What will be delivered

✅ Clear scope ensures:
- Alignment with stakeholders
- Controlled work boundaries

---

## 🔹 2. What Is Scope Creep?

- **Scope creep** = uncontrolled growth of project scope

### Common Example:
- Stakeholder casually requests:
  - “Can you add this feature?”
- No adjustment to:
  - Budget ❌
  - Schedule ❌

👉 Result:
- Increased workload without proper planning

---

## 🔹 3. Causes of Scope Changes

### ✅ 3.1 Stakeholder Requests
- New ideas or requirements as project progresses

---

### ✅ 3.2 Unclear Scope Definition
- Requirements not clearly defined during planning

---

### ✅ 3.3 Assumptions
- Customer assumes:
  - Project team already understands needs

---

### ✅ 3.4 Uncertain Requirements
- Customer doesn’t fully know what they want

---

## 🔹 4. Preventing Scope Creep

### ✅ Best Practice:
- Clearly define scope during **planning phase**

### Include:
- Scope statement
- Detailed requirements
- Boundaries of project

---

## 🔹 5. Managing Scope During Execution

## ✅ 5.1 Reset Expectations
- If requests are out of scope:
  - Refer to **scope statement**

### Example:
- “This request is not part of the approved scope.”

---

## ✅ 5.2 Link Changes to Time and Cost
- Explain impact:
  - Schedule changes
  - Budget increases

✅ Outcome:
- Stakeholders understand trade-offs

---

## ✅ 5.3 Enforce Change Management Process
- No informal changes allowed

### Rule:
- All requests must:
  - Go through **formal change process**

✅ Keeps scope controlled

---

## ✅ 5.4 Clarify Scope if Needed
- If scope was unclear:
  - Work with customer to:
    - Redefine scope
    - Set clear boundaries

---

## ✅ 5.5 Consider Agile Approach
- If requirements are unclear or evolving:

### Use Agile:
- Iterative development
- Continuous refinement of scope

✅ Benefit:
- Flexibility and adaptability

---

## 🔹 6. Warning Signs of Scope Creep

- Frequent informal requests
- Increasing workload without timeline changes
- Team confusion about deliverables
- Stakeholders bypassing formal process

---

## 🔹 7. Impact of Uncontrolled Scope

- Project delays
- Budget overruns
- Resource overuse
- Reduced quality

---

## 🔹 8. Best Practices for Scope Management

- Define scope clearly at the beginning
- Communicate scope to all stakeholders
- Use formal change control
- Track all scope changes
- Educate stakeholders on impact

---

## 🔹 ✅ Key Takeaways
- Scope creep = uncontrolled addition of work
- Clear scope definition prevents confusion
- All changes must go through **change management**
- Always link scope changes to:
  - Time
  - Cost
- Agile is useful when scope is uncertain
- Effective scope control is critical for project success

---
