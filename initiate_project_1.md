## Overview of Project Initiation

Initiating a project is about gaining **formal commitment to move forward**.

In practical terms, initiation is when:
- The **customer or sponsor approves the project**
- The project receives a **go‑ahead to begin planning**

Without clear initiation, projects risk starting without alignment, authority, or shared understanding.

---

## Role of the Project Manager During Initiation

### Assignment of the Project Manager

- Typically, the **first step** in initiation is assigning a project manager
- The project manager guides the project through the remaining initiation activities

In some organizations:
- The project manager is assigned **after** the project is approved

If this happens:
- Review all initiation activities carefully
- Identify gaps or unfinished work
- Complete any missing initiation steps before moving forward

---

## Purpose of Initiation

The sponsor or customer must make an **informed decision** about whether to approve the project.

To support that decision, initiation focuses on **defining the project clearly**.

Key questions include:
- What problem is the project solving?
- What opportunity is being pursued?
- What does success look like?

---

## Defining the Project

During initiation, you gather and document information such as:
- Project objectives
- High‑level requirements
- Expected deliverables
- Constraints and assumptions
- Initial risks

This information forms the **initial project definition**, which becomes the foundation for planning.

---

## The Project Charter

Once the project definition is established, you prepare the **project charter**.

### Purpose of the Project Charter
The project charter:
- Formally **authorizes the project**
- Grants authority to the project manager
- Establishes alignment between stakeholders

### Typical Charter Content
- Project purpose and objectives
- Summary of scope
- High‑level risks and assumptions
- Key stakeholders
- Project manager authority

The charter marks the official start of the project.

---

## Organizing and Storing Project Data

Projects generate a large volume of documents and information that are critical for:
- Current project success
- Improving future projects

A **Project Information System (PIS)** helps organize, store, and retrieve this information efficiently.

---

## Preparing a Project Information System

Before starting a project—or improving an existing system—do the following:

- Identify the information you want to extract from the system
- Assign a **system administrator** to ensure consistency and accuracy

---

## Duties of a Project Information System Administrator

Typical responsibilities include:
- Checking documents in and out with proper labeling and categorization
- Managing **version control** to prevent outdated documents from being used
- Monitoring document and data requests
- Updating categories when needed
- Defining effective keywords for fast retrieval
- Maintaining data backup and security processes
- Preparing **project history packets** to support future projects

---

## Categorizing Project Information

Categories and labels improve retrieval efficiency.

Common categorization approaches include:
- Project categories (e.g., product or service type)
- Project lifecycle (Agile, Waterfall, Hybrid)
- Stakeholders or customers
- Risks that occurred
- Types of lessons learned

The categories you choose should align with the information you want to reuse later.

---

## Approaches to Project Information Systems

Project information systems range from simple to advanced.

### Manual Approaches
- Organized computer folders or paper filing systems
- Inexpensive and effective if well maintained
- Limited support for:
  - Cross‑referencing
  - Version control

### Project Management Software
- Examples include:
  - Microsoft Project
  - Monday.com
  - Zoho Projects
- Supports electronic storage, tagging, and retrieval
- Improves consistency and accessibility

### Repository Management Software
- Automates version control and check‑in/check‑out
- Often designed for specific information types
- Examples include:
  - Archiva
  - Sonatype Nexus
  - GitHub
  - Cloudsmith

---

## Value of a Well‑Designed Project Information System

A strong project information system:
- Helps teams find information quickly
- Reduces wasted time
- Supports better decision‑making
- Preserves lessons learned
- Prevents repeating past mistakes
- Encourages reuse of successful practices

It is both a **productivity tool for current projects** and a **knowledge asset for the organization**.

---

## Key Takeaway

Project initiation establishes authority, clarity, and alignment.  
Defining the project clearly, approving it through a project charter, and organizing information effectively lays the foundation for successful planning, execution, and future learning.

## What Are Project Stakeholders?

As a project manager, you must understand **who has a stake in the outcome of your project**.  
These individuals or groups are called **stakeholders**.

Stakeholders include anyone who:
- Can influence the project
- Is involved in project work
- Is affected by the project’s outcomes

Understanding stakeholders is critical to project success.

---

## Why Identifying Stakeholders Matters

For each stakeholder, a project manager needs to understand:
- **Expectations**: What they want from the project
- **Contributions**: How they support or influence the project
- **Interest**: How much they care about the project’s outcomes
- **Influence**: How much power they have to affect decisions

This knowledge allows you to:
- Build relationships with influential stakeholders
- Manage expectations proactively
- Increase satisfaction with project results

Identifying stakeholders is the **first and most important step**.

---

## Major Stakeholder Roles

---

## Project Customer

The **project customer** is the person or group with the problem the project is intended to solve.

### Case Study Example: Hospital Scheduling Project
- The hospital’s **Chief Operating Officer (COO)** oversees scheduling
- Therefore, the COO is the **project customer**

### What the Customer Contributes
The customer plays three critical roles:

1. **Funding**
   - The customer controls or is accountable for the project budget  
   - In the hospital example:
     - The CEO oversees the overall hospital budget
     - The COO controls the budget for the scheduling project

2. **Defining What the Project Will Do**
   - The customer has significant influence over project scope and requirements

3. **Approving Deliverables**
   - The customer approves deliverables throughout the project lifecycle

---

## Project Sponsor

The **project sponsor** is someone who:
- Wants the project to succeed
- Has enough authority to help ensure success

Sponsors are often senior leaders or executives.

### Case Study Example
- In the scheduling project:
  - The **COO** is also the **project sponsor**

### Sponsor Responsibilities
A project sponsor can:
- Champion the project
- Help prioritize objectives
- Intervene with unsupportive stakeholders
- Suggest improvements to the project plan

The sponsor’s authority makes them a powerful ally for the project manager.

---

## Functional or Aligned Managers

**Functional managers**:
- Lead departments (e.g., IT, Facilities, Nursing)
- Are accountable for departmental performance
- Manage the staff you need for your project

Their cooperation is essential because:
- They control resource availability
- Project team members often report to them

---

## Project Team Members

Team members are also stakeholders.

Why this matters:
- Their work directly affects project outcomes
- Their performance on the project may affect their careers
- Their job security or future assignments may depend on project success

Project managers must:
- Communicate clearly with team members
- Balance project needs with functional responsibilities

---

## Affected and Influencing Stakeholders

Some stakeholders:
- **Affect the project**
- **Are affected by the project**
- Or both

### Case Study Example
- The scheduling project affects how hospital facilities are assigned
- Therefore, the **Facilities Department** is a stakeholder

These stakeholders may not work directly on the project, but their:
- Processes
- Tools
- Daily work
can be impacted by project outcomes.

---

## Managing Stakeholders Effectively

To keep stakeholders satisfied, project managers must:
- Identify all stakeholders early
- Understand their roles and concerns
- Assess their influence and interest
- Communicate appropriately with each group

---

## Key Takeaway

Stakeholders influence whether a project succeeds or fails.  
The first step in managing stakeholders effectively is **knowing who they are**.

By identifying stakeholders early and understanding their expectations, influence, and interests, a project manager can:
- Build strong relationships
- Avoid surprises
- Increase the likelihood of successful project outcomes

## Overview

Identifying stakeholders is only the first step.  
To manage them effectively, a project manager must **analyze stakeholders** to understand:
- Who is most important
- What motivates each stakeholder
- How to work with them effectively

This information is captured in a **stakeholder analysis document**.

---

## Purpose of a Stakeholder Analysis

A stakeholder analysis helps you:
- Organize information about stakeholders
- Understand their role in the project
- Prioritize your time and communication
- Reduce surprises and resistance

Stakeholder analysis is **ongoing**.  
You continue refining it as you define the project and learn more.

---

## Stakeholder Analysis Document

The stakeholder analysis document (often a spreadsheet) is used to:
- Record stakeholder details
- Track influence and interest
- Capture expectations and contributions

You update this document throughout project initiation and planning.

---

## Key Elements to Analyze

---

## 1. Stakeholder Background and Role

Start by documenting:
- Department, business unit, or company
- Stakeholder’s position in the organization
- How they are connected to the project

This establishes **where they fit** and the perspective they are likely to bring.

---

## 2. Influence Network

Determine:
- Who the stakeholder listens to
- Who influences their decisions

This is valuable when:
- A stakeholder is resistant
- You need help negotiating or persuading
- You want advice on how to work with them

You can:
- Ask that influencer for guidance
- Ask them to help directly

---

## 3. Stakeholder Objectives

Identify:
- Which **project objectives** the stakeholder cares about
- How they **prioritize** those objectives

This allows you to:
- Direct issues to the right stakeholders
- Resolve conflicts more efficiently
- Tailor communication based on what matters most to them

---

## 4. Influence and Interest

Categorize each stakeholder by:
- **Influence** (ability to affect project decisions)
- **Interest** (level of concern about project outcomes)

This categorization helps you:
- Prioritize stakeholders
- Decide how much time and effort to spend with each group
- Avoid over‑ or under‑communicating

---

## 5. Stakeholder Contributions

Document what each stakeholder contributes, such as:
- Funding
- Decision‑making authority
- Subject matter expertise
- Resources
- Approval of deliverables

Knowing contributions helps you:
- Set expectations
- Know who to contact for support or decisions
- Anticipate dependencies

---

## Tools for Practice

For practice:
- Use the **stakeholder analysis spreadsheet** included with the course
- Fill it out for one of your own projects

The exercise files also include:
- A completed stakeholder analysis for the **hospital scheduling project**

These examples show how stakeholder analysis works in real situations.

---

## Why Stakeholder Analysis Matters

Managing stakeholders without understanding them can be:
- Overwhelming
- Time‑consuming
- Ineffective

Stakeholder analysis provides clarity by revealing:
- What stakeholders want
- How they affect the project
- How best to engage them

---

## Key Takeaway

Stakeholder analysis is a **core activity during project initiation**.

By systematically analyzing stakeholders, project managers can:
- Build stronger relationships
- Focus effort where it matters most
- Anticipate challenges
- Increase the likelihood of project success

Knowing **who your stakeholders are is not enough**—you must understand them.

## Overview

The **first step toward project success** is understanding what the customer truly wants.  
This starts with identifying the **project goal**.

The project goal describes the **end result** the project is intended to deliver.  
It represents the problem to be solved or the opportunity to be pursued and **drives every decision** made throughout the project.

---

## What Is a Project Goal?

A project goal:
- Describes the desired outcome of the project
- Focuses on *why* the project exists, not how it will be done
- Provides direction for planning, decision‑making, and execution

A clear project goal helps:
- Gain stakeholder buy‑in
- Align the project team
- Measure success

---

## Start with a Problem Statement

Before defining the goal, you first create a **problem statement**.

### Purpose of the Problem Statement
The problem statement:
- Clearly defines the problem or opportunity
- Explains the motivation for the project
- Keeps the focus on *why* the project is needed

### Best Practices
- Keep it short and focused  
- One sentence is ideal
- Avoid describing solutions

---

## Avoid Jumping to Solutions

People often jump straight to solutions, but:
- A solution describes the **end result**
- The problem statement explains the **reason** for the project

To move from a solution back to the real problem, ask:

> **Why?**

Do not hesitate to ask *why* more than once.  
This helps uncover root causes and clarifies true project objectives.

---

## Case Study Example: Hospital Scheduling Project

### Initial Solution Statement
> “The hospital wants to implement a new scheduling system.”

This describes a solution — not the problem.

---

### Asking “Why?”

1. **Why do we need a new scheduling system?**  
   - Hospital resources are either overloaded or sitting idle.

2. **Why does that happen?**  
   - Procedures require specific equipment, qualified staff, and suitable rooms.

3. **What’s missing today?**  
   - The current system does not coordinate all these elements together.

As a result:
- Equipment may be scheduled when no technician is available
- Rooms may not support the scheduled equipment
- Grants and donations are available to improve these issues

---

## Example Problem Statement

> “Hospital resources are not being utilized efficiently because scheduling does not ensure that required equipment, staff, and facilities are available. Funding is available to address scheduling issues.”

This statement:
- Defines the core problem
- Explains why it matters
- Notes the opportunity to act

---

## Defining the Project Goal

With a clear problem statement, you can define the **project goal**.

### Characteristics of a Good Project Goal
- Simple and easy to understand
- Focused on outcomes, not tasks
- Aligned with the problem statement
- Useful for guiding decisions and gaining buy‑in

---

## Example Project Goal

> “The project will deliver scheduling improvements so hospital resources can be scheduled efficiently. The project will take advantage of available funding for productivity and technology enhancements.”

This goal:
- States the intended outcome
- Links directly to the problem
- Provides a clear direction for the project team

---

## Why the Project Goal Matters

A well‑defined project goal:
- Guides scope definition
- Helps resolve conflicts
- Aligns stakeholders
- Keeps the team focused on the right outcome

Knowing **where you are going** is the best way to start a project.

---

## Practice Exercise

To apply these concepts:
- Write a **problem statement** for your own project
- Then write a corresponding **project goal**
- Use the template provided in the exercise files

---

## Key Takeaway

Project success starts with clarity.

By clearly defining:
- The **problem statement**, and
- The **project goal**

you establish a strong foundation that guides the project from initiation through completion.

## Overview

Once the **project goal** is defined, the next step is to break it down into **detailed project objectives**.

Project objectives:
- Clarify what must be achieved
- Help define **project scope**
- Influence the **project approach**
- Establish **success criteria**

Well‑defined objectives translate a high‑level goal into concrete outcomes.

---

## Types of Project Objectives

Projects typically include several categories of objectives.

---

## 1. Business Objectives

Business objectives support the organization’s **mission and strategy**.

Examples:
- Increase market share to 25%
- Deliver world‑class healthcare
- Improve customer experience or service reach

These objectives justify why the project matters to the organization.

---

## 2. Financial Objectives

Financial objectives focus on **money‑related outcomes**.

Examples:
- Increase revenue by 15%
- Reduce operating costs by 10%
- Improve return on investment (ROI)

These objectives help stakeholders assess the project’s economic value.

---

## 3. Quality Objectives

Quality objectives specify **how good the results must be**.

Examples (hospital context):
- Reduce staph infections by 80%
- Reduce patient readmittance by 75%
- Improve patient satisfaction ratings

Quality objectives ensure results meet required standards, not just completion.

---

## 4. Technical Objectives

Technical objectives define **technical requirements or capabilities**.

Examples:
- Mobile medical equipment usable in the hospital, ambulances, and remote sites
- System compatibility requirements
- Performance or reliability specifications

These objectives are similar to technical specifications in engineering projects.

---

## 5. Performance Objectives

Performance objectives act as a **catch‑all category** and often focus on timing or constraints.

Examples:
- Complete the project by a specific date
- Use government grant funding before expiration
- Meet regulatory or compliance deadlines

In the hospital project:
- Government grants must be used before they expire

---

## Documenting Objectives Using SMART Criteria

Clear objectives are best written using **SMART criteria**.

---

## Specific

- Clearly state what must be achieved
- Avoid vague language

Example:
- ✅ Increase revenue by 15%
- ❌ Improve revenue

Specific objectives reduce misunderstanding.

---

## Measurable

- Objectives must be measurable so success can be evaluated

Examples:
- Percentage decrease in readmittance
- Revenue increases
- Survey-based measures, such as a minimum patient satisfaction rating of 4.5 stars

If you cannot measure it, you cannot confirm achievement.

---

## Achievable (Realistic)

- Objectives must be achievable with available resources
- Challenging goals can motivate teams
- Unrealistic goals may cause disengagement

Objectives should stretch the team—but still be possible.

---

## Time‑Related

- Objectives must include a clear timeframe or deadline

Example:
- Government grants must be used by **December 31, 2020**

Time-based objectives prevent projects from dragging on indefinitely.

---

## Validating Objectives with Benefit Analysis

After defining the project goal and objectives, you work with stakeholders to perform a **benefit analysis**.

The purpose is to:
- Confirm alignment with organizational mission and strategy
- Validate expected business value
- Ensure the project is worth pursuing

---

## Choosing a Project Strategy

There are often **multiple ways to achieve** project objectives.

Strategy selection:
- Begins with brainstorming by a small group familiar with the project
- Involves evaluating how well each strategy supports the objectives

Typically:
- Senior project managers make the final strategy decision
- Objective alignment is a key decision criterion

---

## Role of Project Objectives

Project objectives:
- Flesh out the project goal
- Define project scope
- Influence approach and strategy
- Establish success criteria

They connect **why the project exists** to **how success will be measured**.

---

## Practice Exercise

The exercise files include:
- A project objectives template
- A description of what the hospital wants to achieve

Use these tools to:
- List the hospital’s objectives
- Categorize them by type
- Practice writing SMART objectives

---

## Key Takeaway

Clear, well‑structured objectives are critical to project success.

By defining objectives across business, financial, quality, technical, and performance areas—and documenting them using SMART criteria—you provide clarity, alignment, and measurable direction for the entire project.

## Overview

Once the project **goal, objectives, and strategy** are defined, the next step is to gather **project requirements**.

Requirements describe **exactly what the project must deliver**.  
Getting requirements right is critical to stakeholder satisfaction and project success.

- Missing requirements lead to dissatisfied stakeholders
- Unnecessary requirements increase cost, time, and complexity

---

## What Are Project Requirements?

Requirements are **detailed descriptions of what the project must produce**.

They define:
- What the solution must do
- What capabilities are required
- What constraints must be respected

Requirements translate project objectives into actionable deliverables.

---

## Example: Hospital Scheduling Project

### Objective
- Reduce rescheduling of procedures by **75%**

### Example Requirements
- The scheduling system must **simultaneously schedule staff, equipment, and facilities** for a procedure
- The system must **search for the next available time slot** when all selected resources are available

These requirements are:
- Necessary
- Specific
- Directly tied to the project objective

They can be expanded later with additional detail.

---

## Common Challenges in Gathering Requirements

Gathering requirements is often difficult because:

- Stakeholders may:
  - Describe requirements incorrectly
  - Provide inconsistent or contradictory information
  - Omit essential needs
  - Include “nice‑to‑have” features
- Non‑stakeholders may try to add requirements
- Stakeholders may resist spending time defining requirements

These challenges require patience and structured techniques.

---

## Techniques for Gathering Requirements

---

## 1. Interviews

- Conduct interviews with the **right stakeholders**
- Come prepared with a clear list of questions
- Useful for gathering detailed, role‑specific requirements

---

## 2. Brainstorming Sessions and Focus Groups

- Bring together representatives from multiple groups
- Encourage discussion about needs and expectations
- Helps surface conflicts early
- Builds **buy‑in** across departments

---

## 3. Observation

- Watch how people perform their day‑to‑day work
- Reveals requirements stakeholders may not articulate
- Especially useful for process‑driven projects

After observation:
- Document findings
- Review them with workers to validate accuracy

---

## 4. Questionnaires and Surveys

- Useful when many stakeholders are involved
- Allows broad input with minimal time commitment

Best practices:
- Design questions carefully
- Avoid questions that lead or bias responses

---

## 5. Document Analysis and Reverse Engineering

- Review existing documentation, reports, or systems
- Analyze current outputs to infer requirements
- Reverse engineer existing products or processes when necessary

Useful when:
- Prior solutions exist
- Documentation is available
- Historical data can inform requirements

---

## Analyzing Initial Requirements

After gathering initial requirements:

- Review them for:
  - Completeness
  - Clarity
  - Consistency
  - Duplication
- Identify:
  - Gaps in information
  - Conflicting requirements

Then:
- Return to stakeholders
- Ask follow‑up questions
- Clarify uncertainties

This process often takes **multiple iterations**.

---

## Documenting Requirements

Once requirements begin to make sense:

- Document them clearly
- Use simple, unambiguous language
- Avoid technical jargon when possible
- Ensure all stakeholders can understand them

---

## Organizing Requirements

Organizing requirements into related categories:
- Reduces duplication
- Highlights conflicts
- Improves readability and traceability

Common categories may include:
- Functional requirements
- Technical requirements
- Performance requirements
- Constraints

---

## Why Requirements Matter

Requirements define what the project delivers.

Poor requirements result in:
- Rework
- Cost overruns
- Schedule delays
- Unsatisfied stakeholders

Good requirements:
- Align expectations
- Guide design and development
- Support acceptance and validation

---

## Practice Exercise

For practice:
- Identify which **requirements‑gathering techniques** you would use
- Consider the number of stakeholders, their availability, and the project complexity
- Apply multiple techniques when necessary

---

## Key Takeaway

Requirements gathering is a **foundational project activity**.

By using multiple techniques, validating assumptions, and clearly documenting requirements, project managers create a solid basis for:
- Scope definition
- Planning
- Solution design
- Stakeholder approval

Getting requirements right is essential to delivering the right project.

---
